### Q001 · 多阶段构建是多个 Dockerfile 吗？

- **日期**：2026-06-07
- **关联**：[docker-multi-stage.md](concepts/docker-multi-stage.md)
- **关键词**：`Docker` `多阶段构建` `FROM`

**问题**：多阶段构建是不是意味着要写多个 Dockerfile 文件？

**答案**：不是。多阶段构建是在**一个 Dockerfile 里写多个 `FROM`**，每个 `FROM` 开始一个新阶段。前面的阶段（builder）负责编译，后面的阶段（production）用 `COPY --from=builder` 只拿走产物，builder 整体丢弃不进最终镜像。`docker build` 命令不变。

**核心收获**：
- 一个 Dockerfile，多个 `FROM`，最终只输出最后一个阶段
- `COPY --from=阶段名` 是阶段间传递产物的唯一通道

### Q002 · 为什么要多阶段构建？

- **日期**：2026-06-07
- **关联**：[docker-multi-stage.md](concepts/docker-multi-stage.md)
- **关键词**：`镜像体积` `安全` `生产部署`

**问题**：多阶段构建比单阶段到底好在哪？

**答案**：三个核心差异——
1. **部署速度**：500MB vs 80MB，推送拉取差 5-6 分钟，紧急回滚时差距巨大
2. **安全面**：生产镜像不含 gcc/make 等编译工具，攻击者拿到 shell 也无法编译提权
3. **运维成本**：镜像仓库存储按 GB 计费，每天构建 10 次，单阶段月吃 150GB，Alpine 只要 24GB

Docker 镜像是分层存储，`rm -rf` 只标记不可见，底层数据仍在。多阶段让 builder 层整体丢弃。

### Q003 · Alpine 构建报错：Connection refused / pecl 无包

- **日期**：2026-06-07
- **关联**：[docker-multi-stage.md](concepts/docker-multi-stage.md)
- **关键词**：`Alpine` `apk` `镜像源` `pecl`

**问题**：Alpine 下载包被墙，`pecl install redis` 报 "No releases exist"

**答案**：
1. 换阿里云镜像源：`sed -i 's/dl-cdn.alpinelinux.org/mirrors.aliyun.com/g' /etc/apk/repositories`
2. Alpine 的 PECL 频道数据为空，先用 `pecl channel-update pecl.php.net` 再 `pecl install`
3. 构建缓存坏状态用 `docker builder prune -f` 清理

**核心收获**：
- `apk` = Alpine 的 apt-get，`--no-cache` = `rm -rf /var/lib/apt/lists/*`
- `--virtual .build-deps` 创建虚拟包组，`apk del .build-deps` 一键卸载

### Q004 · 为什么用 Compose 而不是 Dockerfile 区分环境？

- **日期**：2026-06-07
- **关联**：[docker-compose-environments.md](concepts/docker-compose-environments.md)
- **关键词**：`dev` `prod` `职责分离`

**问题**：为什么用 docker-compose 区分环境，而不是在 Dockerfile 里做？

**答案**：Dockerfile 负责**编译时**（怎么造镜像），Compose 负责**运行时**（怎么跑镜像）。dev/prod 的差异——端口映射、display_errors、OPcache 开关、资源限制——全是运行时配置。用 Dockerfile 区分需要构建两个镜像，违背"一次构建处处运行"。正确做法：同一镜像 + 不同 Compose 覆盖文件。

### Q005 · `docker history` 的 `<missing>` 是什么？

- **日期**：2026-06-07
- **关联**：[docker-multi-stage.md](concepts/docker-multi-stage.md)
- **关键词**：`docker history` `镜像分层` `BuildKit`

**问题**：`docker history` 里的 `<missing>` 和 SIZE 列怎么看？

**答案**：
- `<missing>` = BuildKit 不持久化中间层 ID，正常现象，不是数据丢失
- SIZE 列之和 > `docker images` 显示的总大小（共享基础镜像层不重复计费）
- 每条 RUN/COPY/ADD = 一个新层，层不可变，删除只在新增层标记隐藏

### Q006 · 为什么 `docker-compose.override.yml` 自动生效？

- **日期**：2026-06-07
- **关联**：[docker-compose-environments.md](concepts/docker-compose-environments.md)
- **关键词**：`override` `自动合并`

**问题**：把 `docker-compose.dev.yml` 改名为 `docker-compose.override.yml` 就生效了？

**答案**：Compose 约定：执行 `docker compose up` 时自动合并当前目录的 `docker-compose.override.yml`。任意命名（如 `.dev.yml`）不会被自动识别，必须 `docker compose -f docker-compose.yml -f docker-compose.dev.yml up` 显式指定。

| 命名 | 自动加载 | 使用方式 |
|------|:--:|------|
| `docker-compose.yml` | ✅ | 永远自动 |
| `docker-compose.override.yml` | ✅ | 自动叠加 |
| `docker-compose.prod.yml` | ❌ | `-f` 显式指定 |

### Q007 · 为什么 Nginx 里写 `fastcgi_pass php:9000` 而不是 IP？

- **日期**：2026-06-07
- **关联**：[docker-networking.md](concepts/docker-networking.md)
- **关键词**：`bridge` `DNS` `容器通信`

**问题**：Nginx 配置里的 `php` 是什么？为什么不是 172.x.x.x？

**答案**：Compose 自定义 bridge 网络内置 DNS，自动把服务名 `php` 注册为域名，解析到 PHP 容器的动态 IP。写服务名而非硬编码 IP，因为容器重建后 IP 会变，但服务名不变。完整链路：浏览器 → localhost:8080 → Nginx:80 → DNS 解析 "php" → PHP-FPM:9000 → DNS 解析 "mysql" → MySQL:3306，全程零硬编码 IP。

### Q008 · `172.19.0.0/16` 和 `172.19.0.1` 是什么？

- **日期**：2026-06-07
- **关联**：[docker-networking.md](concepts/docker-networking.md)
- **关键词**：`子网` `网关` `docker network inspect`

**问题**：`docker network inspect` 看到的这些 IP 分别代表什么？

**答案**：
- `172.19.0.0/16`：网络地址范围，`/16` = 前 16 位网段号 + 后 16 位容器 IP（共 65534 个）
- `172.19.0.1`：网关 IP，所有容器的默认路由指向这里，容器访问外网必经此地址
- 可验证：`docker exec -it myapp-nginx ping php`（能通，DNS 自动解析）

### Q010 · Jenkins 插件安装大量失败

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`Jenkins` `插件` `镜像源` `GFW`

**问题**：安装推荐插件时 38 个全部下载失败，日志显示 `Failed to download from updates.jenkins.io`

**答案**：Jenkins 官方更新中心在国内被墙。切到清华镜像源：
```bash
docker exec myapp-jenkins sh -c 'cat > /var/jenkins_home/hudson.model.UpdateCenter.xml << EOF
<?xml version="1.1" encoding="UTF-8"?>
<sites>
  <site>
    <id>default</id>
    <url>https://mirrors.tuna.tsinghua.edu.cn/jenkins/updates/update-center.json</url>
  </site>
</sites>
EOF'
```

或者在网页 `Manage Jenkins → Plugins → Advanced → Update Site` 手动改。

**核心收获**：
- Jenkins 所有插件从更新中心 JSON 里取下载地址，只改 JSON URL 不够
- 清华镜像会 rewrite JSON 里的下载地址到国内，完整方案
- `down -v` 清数据重来比逐个重装快

### Q011 · Jenkins 自定义镜像里 Docker CLI 怎么装

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`Docker CLI` `Linux` `Dockerfile`

**问题**：Jenkins 容器里要执行 `docker` 命令，直接挂宿主机的 docker.exe 行吗？

**答案**：不行。Windows 的 `docker.exe` 是 PE 格式，Linux 容器无法执行。正确做法：在 Dockerfile 里下载 Linux 版 Docker CLI 静态二进制：
```dockerfile
RUN curl -fsSL "https://mirrors.tuna.tsinghua.edu.cn/docker-ce/linux/static/stable/x86_64/docker-27.3.1.tgz" \
    -o /tmp/docker.tgz && \
    tar -xzf /tmp/docker.tgz -C /usr/local/bin --strip-components=1
```

`/var/run/docker.sock` 挂进去让 CLI 连接宿主机 Engine，容器本身不跑 Docker Engine。

**核心收获**：
- Docker CLI ≠ Docker Engine，CLI 只是客户端
- 跨平台二进制不通用，必须下载目标平台的版本
- Docker-in-Docker（DinD）是另一个方案，但重量级，一般不需要

### Q012 · Git Bash 执行 `docker exec -it` 报 TTY 错误

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`mintty` `TTY` `winpty` `Git Bash`

**问题**：`docker exec -it myapp-jenkins sh` 报 `the input device is not a TTY`

**答案**：Git Bash 基于 mintty，不支持 `-it` 的 TTY 分配。两个方案：
1. 加 `winpty` 前缀：`winpty docker exec -it myapp-jenkins bash`
2. 不需要交互时直接去 `-it`：`docker exec myapp-jenkins cat /path/to/file`

**核心收获**：
- `-it` = `-i`（stdin）+ `-t`（TTY），非交互场景两都不需要
- mintty 的 TTY 实现和 Docker 期望的不兼容
- PowerShell / CMD 没有这个问题

### Q013 · Jenkins 初始化密码在哪儿

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`initialAdminPassword` `Jenkins` `初始化`

**问题**：`docker exec cat /var/jenkins_home/secrets/initialAdminPassword` 文件不存在

**答案**：两种可能：
1. Jenkins 还没初始化完——等 `docker logs` 看到 `Jenkins is fully up and running`
2. 已经初始化过了——密码被 Jenkins 删了（安全机制），需要重置或在日志里找

**找密码最快方式：**
```bash
docker logs myapp-jenkins | grep -A5 "initialAdminPassword"
```

日志里有个带边框的大框，密码就在里面。

**核心收获**：
- initialAdminPassword 是一次性的，创建完 admin 用户后 Jenkins 会删掉
- 忘记密码：停容器 → 删 `$JENKINS_HOME/secrets/` → 重启，会重新生成
- 日志是最可靠的信息来源

### Q014 · Jenkins 里 docker compose 命令找不到

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`docker-compose` `Compose插件` `静态二进制`

**问题**：Pipeline 里 `docker compose up -d` 报 `unknown shorthand flag: 'd' in -d`

**答案**：静态安装的 Docker CLI 不包含 Compose 插件。`docker compose`（空格）需要 CLI 插件支持。解决办法：下载独立 `docker-compose` 二进制（带连字符）：
```bash
# 宿主机下载
curl -fsSL "https://ghproxy.net/https://github.com/docker/compose/releases/download/v2.29.1/docker-compose-linux-x86_64" \
    -o /tmp/docker-compose-linux
docker cp /tmp/docker-compose-linux myapp-jenkins:/usr/local/bin/docker-compose
docker exec myapp-jenkins chmod +x /usr/local/bin/docker-compose
```

**核心收获**：
- `docker compose`（v2 插件）≠ `docker-compose`（独立二进制），功能一样但调用方式不同
- 静态安装的 Docker CLI 没有插件目录，必须用独立版 compose
- Pipeline 脚本里的命令要用 `docker-compose`（带连字符）

### Q015 · Jenkins 部署时 bind mount 报 not a directory

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`bind mount` `docker.sock` `路径翻译` `Docker-out-of-Docker`

**问题**：Jenkins 构建时 docker-compose 报 `error mounting "/var/jenkins_home/workspace/.../www.conf" → not a directory`

**答案**：Docker-out-of-Docker 的经典问题。Jenkins 容器通过 `/var/run/docker.sock` 调用宿主机 Docker Engine——CLI 在容器里跑，Engine 在宿主机上。compose 文件里的相对路径 `./docker/php/www.conf` 被解析为容器内的绝对路径 `/var/jenkins_home/workspace/...`，但宿主机上根本没有这个路径。

**解决：** 创建独立 compose 文件，完全去掉 bind mount。PHP 配置和 Nginx 配置都通过 Dockerfile `COPY` 打入镜像。

**核心收获**：
- Docker socket 挂载让容器能调宿主机 Docker，但文件路径仍是容器视角，Engine 解析不了
- 部署场景下 bind mount 不靠谱，配置全部打入镜像更可移植
- 开发用 bind mount（方便改），部署用 image-only（路径干净）

### Q016 · Jenkins 部署后访问 8080 显示 File not found

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`Dockerfile` `COPY` `develop.watch`

**问题**：Jenkins 部署成功后访问 `localhost:8080` 显示 `File not found`

**答案**：Dockerfile 里只 COPY 了 PHP 配置文件（php.ini、www.conf），没有 COPY 应用代码 `public/`。开发环境靠 `develop.watch` 把代码同步进容器，Jenkins 部署用的 compose 去掉了 watch，镜像里没代码。

**解决：** Dockerfile 补上：`COPY ./public /var/www/html/public`

**核心收获**：
- 开发依赖 `develop.watch` 的同步能力，生产镜像必须自包含所有代码
- 构建出的镜像应该不依赖任何外部挂载就能跑
- 每次改完 Dockerfile 记得 push，Jenkins 构建时 `--build` 才会拉最新 Dockerfile

### Q017 · Pipeline 脚本保存时提示需要审批

- **日期**：2026-06-14
- **关联**：[jenkins-setup.md](concepts/jenkins-setup.md)
- **关键词**：`Script Approval` `安全机制` `Pipeline`

**问题**：每次改 Pipeline 脚本保存时都提示 "The script is not approved"

**答案**：这是 Jenkins 安全设计——Pipeline 脚本可以执行任意 shell 命令，每次修改都必须管理员手动审批。流程：保存 → `Manage Jenkins → In-process Script Approval → Approve` → 构建。

**核心收获**：
- 每次脚本改动（哪怕一个字符）都会触发审批
- 后续可以把 Pipeline 改成 `Jenkinsfile`，脚本存在 Git 仓库里、Jenkins 拉取执行，避免手动贴脚本
- 本地开发环境可以装 `Permissive Script Security` 插件跳过审批，但不推荐

## 🤖 MetaBot × 飞书：移动端 AI 编码

### Q018 · MetaBot 是什么？和直接在终端用 Claude Code 有什么区别？

- **日期**：2026-06-15
- **关联**：[metabot-feishu.md](concepts/metabot-feishu.md)
- **关键词**：`MetaBot` `飞书` `Claude Code` `Agent SDK`

**问题**：MetaBot 是干什么的？我已经有 Claude Code 终端了，为什么还要 MetaBot？

**答案**：MetaBot 是一个桥接服务，把飞书/TG/微信的消息翻译成 Claude Code Agent SDK 调用。核心价值不是替代终端，而是**移动化 + 异步化**：

1. **手机编码**：坐地铁、排队、躺床上时打开飞书就能指挥 Claude 写代码
2. **Agent 团队**：可以配多个 Bot（前端 bot、后端 bot、运维 bot），各干各的不打架
3. **持久会话**：`METABOT_PERSISTENT_EXECUTOR` 让后台任务跨消息存活——发一条 `/background 重构 controller`，关掉飞书，半小时后打开看结果
4. **无需公网 IP**：飞书走 WebSocket 长连接，在 NAT/防火墙后就能跑

MetaBot 不代理 AI 请求——Claude CLI 仍在本地运行，MetaBot 只是让飞书消息能驱动它。

**核心收获**：
- MetaBot = IM ↔ Claude Agent SDK 的翻译层，不替代 Claude
- 飞书/TG/微信三条通道，同一套引擎
- 安装：`curl ... | bash`，5 分钟上手

### Q019 · 飞书应用接入 MetaBot 全流程是怎样的？

- **日期**：2026-06-15
- **关联**：[metabot-feishu.md](concepts/metabot-feishu.md)
- **关键词**：`飞书开放平台` `长连接` `WebSocket` `应用发布`

**问题**：在飞书上用 MetaBot，从零到能聊天，需要哪些步骤？

**答案**：飞书不像 Telegram 给个 bot token 就完事，需要 6 步：

1. **创建应用**：open.feishu.cn → 创建自定义应用 → 添加「机器人」能力
2. **配权限**：`im:message` + `im:message:readonly` + `im:resource` + `im:chat:readonly`
3. **配事件**：选「长连接」→ 添加 `im.message.receive_v1` 事件（**必须先启动 MetaBot 再保存，因为飞书会立即验证 WebSocket**）
4. **发布应用**：创建版本 → 发布（版本号如 `1.0.0`）
5. **配置 MetaBot**：把 App ID 和 App Secret 写入 `.env` 或 `bots.json`
6. **启动测试**：`npm run dev` → 飞书搜索机器人名 → 发消息

**核心收获**：
- 飞书用 WebSocket 长连接，不回调 HTTP，不需要配 Webhook URL
- 事件订阅必须先启动服务——飞书保存时立即验证连接
- 发布后搜不到机器人？退出飞书重新登录（客户端缓存）
- 权限和事件配置错了会静默收不到消息，检查 `docker logs` 或 MetaBot 日志
